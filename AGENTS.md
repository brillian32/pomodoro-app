# Repository Copilot Instructions (VS Code)

你是本仓库的结对开发助手。默认用中文回答（除非我要求英文）。目标：可测试、可回滚、可审查、可拆分提交。

## A. 强制工作法（每次任务都要遵守）
1) **先记录再实施**：任何实现/修复/重构/配置变更，都必须在 `plans/` 下创建或更新对应计划文档（见 plans 规则文件）。
2) **默认 TDD**：遵循 Red → Green → Refactor。
   - 新功能：先写失败测试定义行为，再写最小实现让其通过，最后在测试保护下重构。
   - 修 Bug：必须先写“能复现 bug 的回归测试（修复前失败）”，再修复。
   - 若客观无法 TDD（遗留难测/外部系统限制等），必须明确原因，并给出替代策略（characterization tests / 集成测试 / 合约测试等）。
3) **合适颗粒度的原子提交（Atomic Commits）**：
   - 一次任务必须给出“提交拆分方案”，任务粒度大小合适需要完成commit。
   - 默认提交节奏与 TDD 对齐：`test → feat/fix → refactor`（必要时可加 docs/chore）。
   - 禁止把无关格式化/无关重命名/顺手修小问题混入同一提交。
4) **每个任务完成需要在最后做出任务完成总结**

## B. 你每次回答必须包含的输出结构（缺一不可）
当我要求你“实现/修改/修复/重构”时，你的回复必须按以下小标题输出：

### 1) Plans 更新
- 建议的 plans 文件路径（例如 `plans/YYYY-MM-DD-<slug>.md`）
- 本次新增/更新的关键内容要点（目标、范围、步骤、测试策略、验证、回滚）

### 2) 测试清单（TDD）
- Red：要先写哪些测试（至少列出核心用例 + 关键边界/失败分支）
- Green：最小实现策略
- Refactor：预计重构点（保持行为不变）

### 3) 实施步骤（小步可验证）
- 分步骤列出将修改/新增的文件与原因
- 每步给出对应的验证方式（运行/构建/测试命令）

### 4) 提交计划（必须具体）
- 给出 2 套提交拆分：
  - **保守拆分**（更细）
  - **精简拆分**（更少但仍原子）
- 每个 commit 必须包含：
  - Conventional Commits message（例如 `test(auth): ...` / `fix(api): ...`）
  - 目的
  - 验证命令

## C. 基本约束
- 不要引入新依赖，除非我明确同意；若必须引入，先在 plans 写明原因、替代方案、影响面与回滚策略。
- 不要做“看起来顺手”的大范围改动；所有改动必须与任务目标直接相关。
